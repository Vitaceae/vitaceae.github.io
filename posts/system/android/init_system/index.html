# Linux kernel

PID 0 最終經由 init_idle_bootup_task() 退化為 idle process(cpu_idle)，負責 idle 排程。
此外產生 PID 1 的 init process 執行 /init，帶起 Android 的入口 main()

kernel/init/main.c

Linux Kernel
    start_kernel()
        ↓           [init thread]       [kthreadd thread]
    rest_init() →   kernel_init() +     kthreadd()
                        ↓                   ↓
                    do_basic_setup()    kernel threads
                        ↓
                    run_init_process()
------------------------↓-------------------------------------------------------
Android             main() →            [Zygote Process]
                        ↓               zygote
                    [Local Process]         ↓
                    ADB                 System Server
                    Service Manager         ↓
                    Surface Flinger     Acrtivity Manager
                    ...                 Package Manager
                                        ...

查看 init process 的 PID:

$ adb shell ps -A | grep init

# Init System: SetupSelinux()

  • selinux.cpp
  • 掛載 SELinux 權限系統
    • /sys/fs/selinux/enforce 檔案 0 代表 permissive，1 代表 enforcing
      permissive mode: 允許任何操作，不存在 MAC(Mandatory Access Control)。若違反權限仍會紀錄到 log，多用於 Engineering Mode。
      enforcing mode: 任何操作都會檢查權限。屬於一般 User Mode 或 User-debug Mode
  • 使用 execv 保留 PID 重新載入 main，執行 SecondStageMain

## AIL, Android Init Language

詳細格式參考 system/core/init/README.md 。

init.rc 包含五種類型宣告:

  • Action
    • 以 Section 形式表達。直到下一個 Section 開頭為止構成完整描述；沒有結束標記。
    • 重複的 Action 會被合併。
    • 以關鍵字 on 開頭的範例 
        on early-init                           # Action 類型宣告; 觸發器為字串 "early-init"
      write /proc/1/oom_score_adj -1000   # Command
    • trigger(觸發器): 可使用字串如 “early-init” 或屬性 property:sys.boot_from_charger_mode=1；以 && 連接
  • Command
    • 常與 Action 併用；實際 Linux 操作命令。
    • builtins.cpp
      GetBuiltinFunctionMap() 定義可用命令 
        static const BuiltinFunctionMap builtin_functions = {
      {"bootchart",               {1,     1,    {false,  do_bootchart}}},
      ...
  };
  • Service
    • 以 Section 形式表達。直到下一個 Section 開頭為止構成完整描述；沒有結束標記。
    • 不允許使用重複的 Service 名稱。
    • 以關鍵字 service 開頭的範例 
        service ueventd /sbin/ueventd           # Service 類型宣告; 包含服務名稱與執行路徑，透過 start 命令執行
      class core                          # Option 修飾服務; 可包含複數敘述
      critical
  • Options
    • Service 的啟動參數與設定
  • Import
    • 關聯其他 .rc 檔 
        import /init.${ro.hardware}.rc

## Parser 解析器

init.cpp 中建立並使用 Parser

    Parser parser = CreateParser(action_manager, service_list);
    parser.ParseConfig(...)

ActionParser, ImportParser, ServiceParser 解析對應 section

  • parser.cpp, parser.h
  • action_parser.cpp, action_parser.h
    • 解析結果存放在 std::unique_ptr<Action> action_
  • import_parser.cpp, import_parser.h
    • 解析結果存放在 std::vector<std::pair<std::string, int>> imports_
  • service_parser.cpp, service_parser.h
    • 解析結果存放在 std::unique_ptr<Service> service_
    Parser::ParseConfigFile() → ParseData() → <Action?>     → ActionParser::ParseLineSection() 將 command 加入 Action
                                            → <Import?>     → ImportParser
                                            → <Service?>    → ServiceParser::ParseLineSection() 將 option 加入 Service

## Trigger 觸發

  • action_manager.cpp
  • QueueEventTrigger() 將 trigger 字串加入 event_queue_
  • QueueBuiltinAction() 建立 Action 物件並附加 command 後加入 event_queue_
  • event_queue_ 宣告在 ActionManager 類別中: std::queue<std::variant<EventTrigger, PropertyChange, BuiltinAction>> event_queue_

init.rc 中 Action 次序決定觸發順序，
而 SecondStageMain() 中的 LoadBootScripts() 決定 .rc 載入順序；相同目錄下經由 Parser::ParseConfigDir() 排序後以升冪順序執行。

    init.rc → import in init.rc → nested import → /system/etc/init/ → /vendor/etc/init/ → /odm/etc/init/
                      ↑                ↓
                      ←    recursive   ←
Path Description
/init.${ro.hardware}.rc hardware vendor 設定
/system/etc/init/hw/init.rc hardware vendor 設定
/system/etc/init Android 系統設定如 SurfaceFlinger, MediaService, logcatd
/vendor/etc/init SoC vendor 設定
/odm/etc/init ODM 設定
/product/etc/init 個別產品設定
